## 摘要
一个持久的交互式世界模型将其运行状态驻留在服务它的GPU上：一个占据数吉字节的注意力缓存，几乎在每次生成步骤中都会重写。该状态无法在交互时间内重新计算，也无法在不改变世界的情况下近似，因此实时会话会固定其设备。这种固定是一个调度问题。WorldMove在以下保证下移动实时会话：目标与源按位相同，否则不安装任何内容。它在同节点内以18.8毫秒的速度重新定位缓存，比保存/加载快101倍。它在100 Gb网络上保持校验和验证的92.1-94.8 Gb/s速率。按照这个速率，缓存大小适合一个交互块内。在迁移一个活跃生成会话时，它在块边界收敛，目标端逐位继续世界。可接纳性条件决定每次移动。移动必须在读取时间窗口内完成，带宽需覆盖状态及其脏率。提升到集群可调度性测试，它控制了一个整合循环，在两个提供商之间执行了48次迁移中的48次按位一致。两个约束是结构性的。按位精确性仅在一个GPU架构的受控配置下存在，因此移动状态是在交互时间内精确保存状态的唯一方式。验证不能隐藏在这个网络的有线链路上。在扇入情况下，接收路径的校验和会以协议时间尺度阻塞传输，而无调度的incast会悄无声息地使接收器崩溃，尽管每个传输的字节保持正确。一个感知incast的接纳控制器在1.4倍负载下保持零未命中，并将过载作为拒绝丢弃。一个无损GPU编解码器拓宽了接纳门，使原始移动无法使用的网络也能使用。我们分别端到端地执行服务循环和移动器。它们在同一个网络上的组合尚未构建。精确状态弹性是传输和验证上的联合调度问题。

## Abstract
A persistent interactive world model keeps its running state resident on the GPU that serves it: a multi-gigabyte attention cache, almost all of it rewritten at every generation step. That state cannot be recomputed in interactive time or approximated without changing the world, so a live session pins its device. The pin is a scheduling problem. WorldMove moves a live session under one guarantee: the destination is bit-identical to the source, or nothing is installed. It relocates the cache in 18.8 ms same-node, 101x faster than save/load. It holds a checksum-verified 92.1-94.8 Gb/s on a 100 Gb fabric. At that rate the cache fits inside one interactive block. Migrating an actively generating session, it converges at a block boundary and the destination continues the world bit for bit. An admissibility condition decides each move. The move must complete inside the readout horizon, over bandwidth that covers the state plus its dirty rate. Lifted to a fleet schedulability test, it governed a consolidation loop that executed 48 of 48 migrations bit-identical across two providers. Two constraints are structural. Bit-exactness survives only inside a controlled configuration of one GPU architecture, so moving the state is the only way to preserve it exactly in interactive time. Verification cannot hide inside the wire on this fabric. Receive-path checksums stall the transport at protocol timescales under fan-in, and unscheduled incast silently collapses a receiver while every delivered byte stays correct. An incast-aware admission controller holds zero misses to 1.4x offered load and sheds overload as rejects. A lossless GPU codec widens the admission gate to fabrics raw motion cannot use. We exercise the serving loop and the mover separately, each end to end. Their composition on one fabric is unbuilt. Exact-state elasticity is a joint scheduling problem over transport and verification.

# 论文详细总结：Stateful Worlds, Stateless Elasticity

## 1. 论文的核心问题与整体含义（研究动机和背景）

*   **核心问题**：交互式世界模型（如实时视频生成）的运行状态是一个巨大的、驻留在GPU上的注意力缓存（可高达数GB）。该缓存在每一次生成步骤中都几乎被完全重写（脏率高），且无法在交互时间内重算或近似。这导致一个关键问题：**GPU设备被用户会话“钉住”**，一个GPU只能服务一个用户，使得服务成本极高（论文引用世界实验室和Decart的案例，成本可达 $72/用户小时）。
*   **研究动机**：打破这种“一人一卡”的昂贵模式，实现有状态会话的、具有弹性的服务。其核心瓶颈并非计算能力（FLOPS），而是**GPU状态的不可迁移性**。
*   **整体含义**：论文将**状态迁移**定义为一个**调度问题**。旨在解决在保持“位精确连续性”（即迁移后的世界必须与未迁移时完全一致）的严格合约下，如何在交互时间（毫秒级）内安全、高效地迁移巨大的、正在被不断重写的GPU状态，从而实现GPU资源的弹性共享和高效利用。
*   **背景**：现有的方法（如LLM的KV缓存迁移、虚拟机迁移、检查点恢复、有损压缩）均无法满足交互式世界模型对“位精确”、“实时”、“不可重算”状态迁移的需求。

## 2. 论文提出的方法论：核心思想、关键技术细节

*   **核心思想**：将交互式世界模型的**状态迁移**视为一个**可调度、受约束的实时任务**。系统（Worldline）提供基础原语WorldMove，并定义了判断迁移是否合法的**可接纳性条件**。
*   **关键技术与细节**：
    1.  **WorldMove**：一个保证“要么目标与源完全一致，要么不安装”的**位精确**缓存迁移器。它通过（展平 -> 传输 -> 校验 -> 恢复）的原子操作完成。
    2.  **可接纳性条件 (Exact-State Admissibility Condition)**：这是本文的核心理论贡献。条件指出，一次迁移是合法的，必须满足两个不等式：
        *   **时间门控 (Deadline Gate)**：迁移完成时间 (`T_migrate`) 必须小于合约规定的可接受的实时性 (Horizon，例如 400ms)。`T_migrate(S) < H`
        *   **带宽与脏率 (Flux Bound)**：网络带宽 (`B_fabric`) 必须能够覆盖状态的传输速度加上其被持续写入（脏）的速度。`B_fabric >= S/H + D`。这个条件确保了在实时迁移（live overlap）模式下，网络可以“跑赢”状态被不断修改的速度。
    3.  **预拷贝 (Pre-copy) 收敛性**：在实时迁移中，由于状态脏率很高（约0.71-0.78），预拷贝（先传所有数据，再传增量）无法收敛，最终必须采用停止-拷贝（stop-and-copy）的方式，在块边界处冻结源端，完成最后的完整状态传输。
    4.  **验证平面 (Verification Plane)**：论文发现，在100Gb高速网络下，**校验和 (CRC) 计算是一个独立的、必须被调度的资源**。如果校验和放在接收路径上，会与传输路径发生耦合，引发协议级别的定时器超时和数据包丢失，导致传输稳定性问题。
    5.  **GPU无损压缩**：对于带宽较低的原始网络，引入GPU上的无损压缩（如nvCOMP的ANS熵编码），可将有效载荷减小约35%，从而扩大可接纳的状态迁移范围。
    6.  **Worldline服务系统**：一个搭载WebServe原语的系统，提供`register_state`， `admit`， `migrate`， `verify`， `commit`等六种API，将状态作为一类合约驱动的对象进行管理。

## 3. 实验设计：数据集 / 场景、Benchmark、对比方法

*   **数据集/场景**：

    *   **验证模型**：`Qwen2.5-0.5B` 文本LLM（用于验证原语对不同类型的KV缓存的适用性）。
    *   **负载生成**：使用了可公开获取的`Azure LLM Inference`和`BurstGPT`的请求轨迹，并结合了游戏会话、物理AI回合等特征来模拟现实负载。
*   **Benchmark**：

    *   **合约基准**：将WebServe与**无迁移的“钉住”模式 (pin-queue)** 和**不考虑截止时间的轮询调度 (blind-RR)** 进行比较。
*   **对比方法**：
    *   对比了多种**有损**状态处理方法，如**量化、驱逐、低秩投影**等，结果表明这些方法都会**“破坏世界”**（导致60-92 dB的信号衰减），证明了在合约下**无损迁移的不可替代性**。
    *   验证了**跨GPU架构（如A100 -> H100）的确定性重新计算会失败**，进一步佐证了移动状态的必要性。

## 6. 论文的主要结论与发现

*   **状态迁移是可行的且高效的**：WebServe在18.8ms内实现了同节点的位精确状态迁移，在100 Gb网络上达到92.1-94.8 Gb/s的校验-验证后传输速率。
*   **验证是第二个调度平面**：在高速网络下，**校验和（验证）的放置位置**是系统稳定性的关键。放在接收路径上会引发`incast`性能崩溃。
*   **`Incast`问题至关重要**：未调度的、多对一的并发全缓存迁移会**悄无声息地**导致接收端性能崩溃（从73 Gb/s降至8 Gb/s），尽管传输的字节都是正确的。这需要专用的`incast`感知准入控制器解决。
*   **GPU无损压缩能拓宽适用边界**：对于带宽较低的原始网络，使用GPU端的无损压缩（如nvCOMP）可以显著减小传输数据量，从而将原本不可接纳的网络（如25 Gbps）变为可接纳。
*   **跨架构位精确性不可行**：在同一个GPU架构内（如H100到H100），可以保证位精确的确定性。但在不同架构之间（如A100到H100），即使软件栈相同，也会在第一步生成时产生不可修复的差异。

## 7. 优点

*   **问题定义清晰且实践性强**：清晰揭示了生成式世界模型服务中的一个关键瓶颈——“设备固定”问题，并提出了精确形式的可接纳性条件。
*   **方法新颖且系统性强**：从状态原语（WorldMove）、调度条件到系统组件（Worldline）和验证平面，构建了一套完整的、面向合约的弹性服务理论，而不仅仅是优化某个传输层。
*   **实验设计严谨且充分**：覆盖了从单个原语、实时迁移到集群incast、跨云平台等多个维度的实验，数据充分。特别是对`incast`问题的发现和验证，具有重要的工程价值。
*   **开箱即用**：论文声明开源了一个可运行的演示，增强了结果的可复现性和可信度。
*   **洞察深入**：指出“验证”是一个独立的调度问题，具有和传输不同的资源特性，是重要的系统级洞察。

## 8. 不足与局限

*   **组件尚未完全集成**：**实时服务循环（与模型交互）和高速线速fabric（100 GbE）上的WorldMove尚未进行联合端到端测试**。迁移测试是在分离的、吞吐量驱动的线速fabric上完成的。
*   **物理隔离假设**：大规模集群的结果基于跨节点模型的模拟，未在真正的数据中心fabric网络上完成。
*   **模型家族验证局限**：主要的定量分析建立在`Matrix-Game-2`一个模型上，对其他模型家族（如仅有动作条件型）只是进行推论和间接测量。
*   **生产环境轨迹缺失**：缺少公开的可重放的世界模型生产轨迹，因此仿真负载是基于LLM轨迹和公共数据拟合的，未在真实世界模型负载下验证。
*   **迁移机制粒度**：迁移原子性是“块边界”。缺乏“子块”粒度、增量编码等更灵活的机制，是一个工程上的局限。
*   **时间窗口限制**：验证的高速数据速率（92-95 Gb/s）只是在线速fabric的“一张快照”上测量的，不能完全代表生产环境下的时间稳定性。
*   **特殊硬件要求**：为了达到高性能，验证和（可选）压缩依赖于NVIDIA GPU和特定的网络硬件（NIC），限制了平台的可移植性。
